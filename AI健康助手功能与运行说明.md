# AI健康助手功能与运行说明

## 1. 当前已具备的功能点

> 以下同时包含“已上线能力”和“本轮文档新增的待开发能力说明”，用于指导下一阶段实现。

## 1.1 会话与问答

1. 支持发起多轮对话（患者自然语言提问）。
2. 支持会话列表展示与会话切换。
3. 支持会话历史消息读取（按 session_id 拉取）。
4. 问答结果会落入会话内存存储（重启服务后会丢失，当前为 V1 轻量方案）。

## 1.2 智能导诊

1. 支持“快速导诊”入口。
2. 导诊输出已按 `AI健康助手需求文档.md` 6.1 的模板严格约束，固定结构为：
   - 结论摘要
   - 风险等级（低/中/高）
   - 建议行动（最多3条）
   - 可选就诊科室
   - 免责声明
3. 命中高危词（如胸痛/呼吸困难等）时，优先给出急诊提示。

## 1.3 多模型路由（千问 + DeepSeek）

1. 高风险问题：优先路由到 Qwen。
2. 常规问题：优先路由到 DeepSeek。
3. 首选模型失败时自动回退到另一模型。
4. API key 通过环境变量/.env 读取，不写死在代码。

## 1.4 流式输出（SSE）

1. `AI Chat` 已支持流式输出（后端 `stream=1`）。
2. 前端逐字显示回答增量，减少“请求超时但后台已处理”的体验问题。
3. 流式结束时返回 done 事件，包含最终 answer/session_id/reply_id。

## 1.5 门诊动作联动

在 AI 页面提供快捷动作按钮：
1. 去挂号（跳转 `/outpatient/register`）
2. 看排队（跳转 `/outpatient/queue`）
3. 去支付（跳转 `/outpatient/payment`）
4. 刷新会话（手动拉取最新会话与消息）

## 1.6 新增功能（本轮规划）

### A. 个性化康复伴行

功能作用：
- 用户先选择指定病历，系统提取该病历字段，再结合用户单独输入提示词，生成：
  1) 用药提醒
  2) 复诊建议
  3) 注意事项说明

核心约束：
- 仅做康复管理建议，不做诊断结论，不改处方。
- 必须严格按照固定输出模板返回，便于前端稳定展示。
- `record_id` 必填；`user_prompt` 为单独输入字段（不与病历字段混写）。

### B. 检查报告“翻译官”

功能作用：
- 用户选择某个病历后，系统提取该病历的“诊断信息 + 检查治疗部分”，再结合用户单独输入提示词做非诊断型解释。
- 将专业术语翻译成大白话，提升患者理解度。

核心约束：
- 仅做信息翻译与解释，不做诊断、不替代医生面诊。
- 必须严格按固定模板输出。
- `record_id` 必填；`user_prompt` 为单独输入字段。

### C. AI诊室（本轮新增）

功能作用：
- 患者描述病情，AI诊室进行多轮问询，收集完整健康信息。
- 最后一轮问询完成后，生成结构化诊断报告和AI诊室咨询小结。

核心约束：
- 仅做健康咨询参考，不做确诊结论。
- 必须严格按6轮问询流程和固定输出模板进行。
- 最终输出包含：症状总结、可能疾病、缓解建议、就诊建议、检查建议、用药建议、注意事项、咨询小结。

问询流程（6轮）：
1. 主诉症状收集 - 询问主要不适部位、时间、持续情况
2. 症状详情追问 - 询问具体表现、严重程度、加重/缓解因素
3. 过往病史询问 - 询问慢性病、住院史、手术史
4. 家族遗传病史 - 询问直系亲属健康状况和遗传倾向
5. 用药情况了解 - 询问当前用药、剂量、过敏史
6. 个人健康状况 - 询问生活习惯、工作环境

## 2. 后端接口清单

前缀：`/api/v2`

### 2.1 已落地

1. `POST /ai/chat`
   - 普通 JSON：返回完整 answer
   - 流式 SSE：`/ai/chat?stream=1`
2. `POST /ai/triage`
3. `GET /ai/sessions`
4. `GET /ai/session/:id/messages`

### 2.2 待新增（本轮功能）

1. `POST /ai/rehab-companion`
   - 输入：`patient_id`、`record_id(必填)`、`user_prompt(单独输入)`
   - 处理：后端只提取该 `record_id` 对应病历字段拼装提示词
   - 输出：`rehab_output`（严格模板）

2. `POST /ai/report-translator`
   - 输入：`patient_id`、`record_id(必填)`、`user_prompt(单独输入)`
   - 处理：后端只提取该 `record_id` 对应病历字段拼装提示词
   - 输出：`translator_output`（严格模板）

3. `POST /ai/clinic/start`（本轮新增）
   - 输入：`session_id`（可选，自动生成）
   - 处理：创建诊室会话，返回第1轮问询内容
   - 输出：`session_id`、`current_step`、`content`

4. `POST /ai/clinic/chat`（本轮新增）
   - 输入：`session_id`、`current_step`、`user_response`
   - 处理：保存用户回答，生成下一轮问询
   - 输出：`current_step`、`step_name`、`progress`、`ai_message`

5. `GET /ai/clinic/session/:id`（本轮新增）
   - 获取诊室会话状态和消息历史

6. `POST /ai/clinic/reset`（本轮新增）
   - 重置诊室会话

路由注册文件：
- `application/server/routers/router.go`

处理文件：
- `application/server/api/v2/ai_assistant.go`

## 5. 运行方式

## 5.1 配置模型密钥

推荐使用：`application/server/.env`

示例：

```env
AI_QWEN_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
AI_QWEN_API_KEY=你的QwenKey
AI_QWEN_MODEL=qwen-plus

AI_DEEPSEEK_BASE_URL=https://api.deepseek.com/v1
AI_DEEPSEEK_API_KEY=你的DeepSeekKey
AI_DEEPSEEK_MODEL=deepseek-chat
```

说明：
1. 后端已支持自动尝试加载：
   - 根目录 `.env`
   - `application/server/.env`
2. `.gitignore` 已忽略上述 .env，避免密钥误提交。

## 5.2 启动后端

在 `application/server` 目录执行：

```bash
go run main.go
```

默认会提供 `/api/v2/*` 接口。

## 5.3 启动前端

在 `application/web` 目录执行：

```bash
npm install
npm run dev
```

访问前端后，进入左侧菜单 `AI健康助手` 使用。

## 6. 常见问题排查

## 6.1 问题：请求超时

表现：页面提示 timeout，但刷新后看到回答。

原因：
- 模型响应慢于前端超时阈值。

已做优化：
1. AI API 独立加长 timeout。
2. 已接入流式输出，前端边收边渲染。

仍异常时检查：
1. 外网是否能访问模型 base_url。
2. API key 是否有效。
3. 服务端日志是否有 4xx/5xx。

## 6.2 问题：无模型回复

检查：
1. `.env` 是否生效（变量名是否正确）。
2. 模型名是否存在（如 qwen-plus / deepseek-chat）。
3. 是否触发了回退（主模型失败后应尝试备模型）。
